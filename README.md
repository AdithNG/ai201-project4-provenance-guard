# Provenance Guard

A backend service a creative-sharing platform can plug in to estimate whether a
piece of submitted text was written by a person or generated by AI. It returns a
confidence score, a plain-language transparency label for readers, and gives
creators a way to appeal a verdict they disagree with. Every decision is written
to a structured audit log.

The goal is not to police creativity. Perfect AI detection is an unsolved problem,
so the system is built to communicate uncertainty honestly and to make it hard to
confidently accuse a human of using AI.

## Setup

```bash
python -m venv .venv
source .venv/Scripts/activate    # Windows Git Bash
pip install -r requirements.txt
```

Create a `.env` file in the repo root (it is gitignored):

```
GROQ_API_KEY=your_key_here
```

Run the server:

```bash
python app.py
```

The API listens on `http://127.0.0.1:5000`.

## API

| Endpoint | Method | Accepts | Returns |
|----------|--------|---------|---------|
| `/submit` | POST | `{ text, creator_id }` | `content_id`, `attribution`, `confidence`, `llm_score`, `stylo_score`, `label` |
| `/appeal` | POST | `{ content_id, creator_reasoning }` | `content_id`, `status: under_review`, `message` |
| `/log` | GET | none | `{ entries: [...] }`, newest first |

`/submit` is rate limited and returns `400` on missing fields, `429` over the limit.
`/appeal` returns `404` for an unknown `content_id`.

Example:

```bash
curl -s -X POST http://127.0.0.1:5000/submit \
  -H "Content-Type: application/json" \
  -d '{"text": "The sun dipped below the horizon...", "creator_id": "test-user-1"}'
```

A full `/submit` response is a single structured object carrying the attribution,
both signal scores, the combined confidence, and the transparency label text:

```json
{
  "content_id": "9982b5f0-9a4e-4ce9-b322-2192646ac3a2",
  "attribution": "likely_human",
  "confidence": 0.32,
  "llm_score": 0.2,
  "stylo_score": 0.501,
  "label": {
    "attribution": "likely_human",
    "headline": "Human: Likely human-written",
    "detail": "Our analysis found this reads as human-written, with the natural variation typical of a person's writing. This is an automated estimate and not a guarantee of authorship.",
    "confidence": 0.32
  }
}
```

## Architecture overview: the path a submission takes

1. A creator sends `text` and `creator_id` to `POST /submit`.
2. The rate limiter checks the caller. Over the limit returns `429` before any work.
3. Input is validated. Empty `text` or `creator_id` returns `400`.
4. The text fans out to two independent detectors:
   - Signal 1, a Groq LLM classifier, returns an AI-likelihood and a short rationale.
   - Signal 2, a pure-Python stylometric analyzer, returns an AI-likelihood from
     measurable properties of the prose.
5. The two scores are combined into one calibrated confidence (AI-likelihood, 0 to 1).
6. The confidence is mapped to an attribution category and a transparency label.
7. The decision, both individual signal scores, and the combined confidence are
   written to a SQLite audit log, and the full result is returned to the caller.

The appeal flow: a creator sends `content_id` and `creator_reasoning` to
`POST /appeal`. The system looks up the original decision, flips its status to
`under_review`, appends an appeal entry to the audit log next to the original
classification, and confirms receipt. There is no automated re-classification; a
human reviewer would pick it up from the queue of `under_review` items.

The full architecture diagram for both flows is in
[planning.md](planning.md) under the `## Architecture` section.

## Detection signals

The system uses two genuinely distinct signals. One reads meaning, the other
measures form. Their weaknesses do not overlap, which is what makes combining them
more informative than either alone.

### Signal 1: Groq LLM classifier (semantic)

- **What it measures:** whether the passage *reads* as human or AI-generated,
  holistically. Tone, phrasing, coherence, and how natural or formulaic it feels.
  Implemented with `llama-3.3-70b-versatile` at temperature 0, prompted to return an
  AI-likelihood from 0 to 1 plus a one-sentence rationale.
- **Why it differs between human and AI:** AI prose tends toward an even register,
  hedged balance, and generic phrasing. A capable model has seen enough of both to
  judge that gestalt.
- **What it misses:** it is a judgment, not a measurement. It is unreliable on very
  short text, can be swayed by formal-but-human writing, can be fooled by lightly
  edited AI, and is not perfectly consistent. On any API error or unparseable reply
  it falls back to 0.5 so a flaky call never produces a confident accusation.

### Signal 2: Stylometric heuristics (structural)

Pure Python, no external libraries. Three sub-scores (1 means AI-like), weighted
inside the signal because burstiness is the most reliable and punctuation the least:

| Metric | What it captures | Maps to AI when | Weight |
|--------|------------------|-----------------|--------|
| Burstiness (std-dev of sentence lengths) | rhythm and variation | variance is low (uniform) | 0.5 |
| Type-token ratio (unique / total words) | vocabulary diversity | very high (little repetition) | 0.3 |
| Punctuation density (marks / words) | punctuation habits | clusters mid (~0.12); extremes read human | 0.2 |

`stylo_score = 0.5*burstiness + 0.3*diversity + 0.2*punctuation`. Texts under three
sentences are treated as low-evidence and the raw score is blended halfway toward 0.5.

- **Why it differs between human and AI:** AI text is statistically more uniform
  (lower sentence-length variance, smoother rhythm). Human writing is more irregular.
- **What it misses:** it is blind to meaning. A repetitive, simple-vocabulary human
  poem can look AI-uniform, and a deliberately varied AI prompt can mimic human
  variance. It is also unstable on short text (see Known limitations).

## Confidence scoring

The two signals are combined with a fixed weighting:

```
confidence = 0.6 * llm_score + 0.4 * stylo_score
```

`confidence` is interpreted as an estimated probability that the text is
AI-generated (0 is clearly human, 1 is clearly AI). The LLM is weighted higher
because the stylometric signal is noisier on short and borderline text.

**What the number means to a user.** 0.5 means the system genuinely cannot tell.
The further from 0.5, the more the label commits. The confidence maps to three bands:

| confidence | attribution | label variant |
|------------|-------------|---------------|
| below 0.35 | `likely_human` | Likely human-written |
| 0.35 to 0.65 | `uncertain` | Uncertain origin |
| above 0.65 | `likely_ai` | Likely AI-generated |

The uncertain band is deliberately wide. On a creative platform, confidently
labeling a human's work as AI is the worst outcome, so borderline cases get an
honest "we are not sure" instead of an accusation.

**How I validated the scores are meaningful.** I ran four deliberately chosen
inputs and confirmed they land where intuition says they should, with the two clear
cases far apart and the two borderline cases in the middle:

| Input | llm_score | stylo_score | confidence | result |
|-------|-----------|-------------|------------|--------|
| Clearly AI (formal essay boilerplate) | 0.70 | 0.65 | **0.682** | likely_ai |
| Clearly human (casual ramen review) | 0.20 | 0.50 | **0.32** | likely_human |
| Borderline (formal human economics) | 0.60 | 0.52 | **0.567** | uncertain |
| Borderline (lightly edited AI) | 0.40 | 0.66 | **0.50** | uncertain |

When a borderline case looked off, I printed both signal scores separately to see
which one was over-committing before adjusting.

### Two example submissions with different confidence

**Higher-confidence case (AI-generated essay), confidence 0.682:**

```json
{
  "attribution": "likely_ai",
  "confidence": 0.682,
  "llm_score": 0.7,
  "stylo_score": 0.654
```

**Lower-confidence case (casual human review), confidence 0.32:**

shows the score varies meaningfully rather than hovering around a constant.

## Transparency label

The label is what a reader sees on the platform. It uses plain language, states
that it is an automated estimate, and points to the appeal path. The text changes
by band, not just the number. All three variants are written out verbatim below.

**High-confidence AI** (confidence above 0.65):

> **AI: Likely AI-generated.** Our analysis found strong signs that this text was
> produced by an AI system rather than written by a person. This is an automated
> estimate, not a certainty, and detection can be wrong. If you believe you wrote
> this yourself, you can appeal this label.

**Uncertain** (confidence between 0.35 and 0.65):

> **?: Uncertain origin.** Our analysis could not confidently tell whether this text
> was written by a person or generated by AI. We are showing this honestly rather
> than guessing. The creator is welcome to add context or appeal.

**High-confidence human** (confidence below 0.35):

> **Human: Likely human-written.** Our analysis found this reads as human-written,
> with the natural variation typical of a person's writing. This is an automated
> estimate and not a guarantee of authorship.

The label is returned by `/submit` as a structured object with `headline`, `detail`,
`attribution`, and `confidence`, so a frontend can render it however it likes while
the exact wording stays consistent.

## Appeals workflow

- **Who can appeal:** the creator of the content (identified by the `creator_id` on
  the original submission).
- **What they provide:** the `content_id` from the `/submit` response and
  `creator_reasoning`, free text explaining why they believe the label is wrong.
- **What the system does:** looks up the content (returns `404` if unknown), flips
  its status from `classified` to `under_review`, appends an `appeal` entry to the
  audit log carrying the reasoning and a timestamp next to the original decision
  (which keeps its confidence and both signal scores), and confirms receipt.
- **No automated re-classification.** A human reviewer would open the queue of
  `under_review` items and see the original text, attribution, confidence, both
  signal scores, the timestamp, and the creator's reasoning.

Example:

```bash
curl -s -X POST http://127.0.0.1:5000/appeal \
  -H "Content-Type: application/json" \
  -d '{"content_id": "PASTE-ID", "creator_reasoning": "I wrote this myself. I am a non-native English speaker and my writing can read as more formal than average."}'
```

## Rate limiting

`/submit` is limited to **10 requests per minute and 100 per day per IP address**,
using Flask-Limiter with in-memory storage.

**Reasoning.** A real creator submits their own work occasionally. Even an active
writer revising and resubmitting a draft rarely needs more than a handful of calls a
minute, so 10 per minute is generous headroom for genuine use while still stopping a
script from hammering the endpoint. The 100 per day ceiling caps sustained abuse: a
person does not legitimately submit a hundred original pieces a day, but a scraper
trying to reverse-engineer the detector or run up Groq API costs would blow past it
quickly. Each `/submit` call costs a Groq request, so the limit also protects the
free-tier budget. The numbers are a starting point a platform would tune against
real traffic.

**Observed behavior.** Sending 12 rapid requests returns `200` until the per-minute
limit is reached, then `429` for the rest:

```
request 1 -> 200
...
request 7 -> 200
request 8 -> 429
request 9 -> 429
...
request 12 -> 429
```

(The cutoff lands before the 10th request in this run because three earlier test
submissions in the same minute already counted against the per-minute window.)

The `429` body:

```json
{
  "error": "rate limit exceeded",
  "limit": "10 per minute;100 per day",
  "message": "Too many submissions. Please slow down and try again later."
}
```

auth; here it is open for documentation and grading. Each entry includes a
timestamp, content id, attribution, the combined confidence, both individual signal
scores, status, and either the LLM rationale (for classifications) or the appeal
reasoning (for appeals).

## Known limitations

**Short text breaks the stylometric signal, especially type-token ratio.** On
passages under a few sentences, the type-token ratio saturates near 1.0 because
almost every word is unique, so the diversity sub-score is pinned high for nearly
all short inputs and stops discriminating. In testing, every sample under about 50
words returned a diversity sub-score of 1.0 regardless of origin. The low-evidence
nudge (blending toward 0.5 under three sentences) limits the damage, but it means
the system leans on the LLM signal alone for short content and should not be trusted
on a haiku or a one-line caption.

**Repetitive, simple-vocabulary human writing reads as AI.** A nursery-rhyme-style
poem or a chant with refrains crushes both sentence-length variance and vocabulary
diversity, which is exactly the structural fingerprint the stylometric signal reads
as AI. A human poet leaning on repetition for effect can be pushed toward the
uncertain or AI band by the form of their own work. The wide uncertain band and the
appeal path are the mitigation, but this is a genuine weakness of using structural
uniformity as an AI signal: deliberate human craft and machine uniformity can look
the same on these metrics.

## Spec reflection

**One way the spec helped.** Writing the confidence model in `planning.md` before
any code, the 0.6/0.4 weighting and the 0.35/0.65 bands, gave the scoring function
concrete targets instead of vibes. When I implemented it I had a ready-made test:
run the four reference inputs and confirm the clear cases land outside the bands and
the borderline cases inside. The thresholds existed before the code, so I was
checking the code against the spec rather than inventing the spec to match whatever
the code happened to do.

**One way the implementation diverged.** The plan assumed I could ask Groq for a
structured JSON assessment and parse it directly. In practice, Groq's strict JSON
mode rejected the model's own output with a 400 error whenever the rationale string
was returned without quotes, throwing away a perfectly good AI-likelihood number. I
dropped strict JSON mode and wrote a lenient parser that tries `json.loads` first
and falls back to a regex that pulls the numeric score out of imperfect output. The
spec's intent (a numeric score plus a rationale) survived; the mechanism for getting
it there changed once I saw how the model actually behaved. I also swapped the emoji
icons in the planned label text for plain text markers so the wording stays
consistent across the API, planning doc, and this README.

## AI usage

This project was built using an AI coding assistant for implementation, working
from the `planning.md` spec and architecture diagram, which is the workflow the
assignment is structured around. Two specific instances:

1. **First detection signal.** I directed the assistant to generate the Groq signal
   function from the Detection Signals section of the spec: take text, return an
   AI-likelihood from 0 to 1 plus a rationale. Its first version used Groq's strict
   `json_object` response format. When I tested it directly, that mode returned a 400
   error because the model emitted an unquoted rationale string. I overrode the
   approach: I removed strict JSON mode and added a two-stage parser (strict
   `json.loads`, then a regex fallback for the numeric score) plus a 0.5 fallback on
   any failure, so a flaky API never produces a confident accusation. The number the
   model was already producing was good; only the parsing needed fixing.

2. **Second signal and scoring.** I directed the assistant to generate the
   stylometric function (burstiness, type-token ratio, punctuation density) and the
   combine logic, with explicit instructions to match the spec's 0.6/0.4 weights and
   0.35/0.65 thresholds. I verified the generated constants against the spec rather
   than trusting them, ran the four reference inputs, and confirmed the clear cases
   landed outside the bands and the borderline cases inside. I also added the
   low-evidence handling for short text (blending toward 0.5 under three sentences),
   which the edge-case analysis called for but the generated draft had left out.
